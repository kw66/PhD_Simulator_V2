# vibe2 重构协作说明

更新时间：2026-03-08

## 作用域
- 本文件作用于 `vibe2/` 目录树。
- 仍需同时遵守所有覆盖本目录的上层 `AGENTS.md`、系统指令、开发者指令和用户当面指令。
- 若规则冲突，优先级按“系统 / 开发者 / 用户当面指令 > 更近的 AGENTS.md > 更远的 AGENTS.md”理解。
- 上层 `AGENTS.md` 已覆盖全局编码与乱码处理规则，本文件不再重复定义，只强调在本项目中必须严格执行。

## 项目定位
- `vibe2/` 当前是一个从零开始的新重构项目。
- 默认目标不是恢复历史多项目工作区，而是重构新的游戏本体。
- 除非用户明确要求，否则不要重新引入“老版副本 + 平台壳 + 辅助工具”的并行结构。

## 当前唯一主线
- 只推进新的游戏本体重构。
- 默认不并行开发平台站、部署壳、统计后台、运营页面、辅助脚本或其他旁支项目。
- 如无新指令，始终把“建立正确的新架构并持续推进实现”视为第一优先级。

## 审计优先于实现
- 在继续实现任何模块之前，必须先尽可能完整地研究旧版对应模块。
- 研究重点必须覆盖：设计目标、真实运行逻辑、关键数值、边界条件、交互口径、与其他系统的耦合点。
- 若旧版经过多轮改动，必须主动甄别：
  - 仍在生效的真实逻辑
  - 已弃用但未删除的冗余逻辑
  - 重复实现、冲突实现、历史残留样式与模块
- 只有在对旧版模块的真实设计足够有把握时，才允许进入该模块的重构实现。

## 禁止编造设计与数值
- 不确定的设计、逻辑、数值、门槛、流程，禁止自行编造。
- 若旧版信息不足以确定真实规则，必须明确标记为“待核定”，不能装作已确认。
- 重构时优先迁移“已确认的真实规则”，不要为了推进速度补写未取证机制。
- 若必须做暂时性占位，必须明确说明这是临时骨架，不得冒充旧版真实规则。

## 禁止偷懒式复制
- 不允许为了图快而直接整段复制旧版代码、旧版模块或旧版目录。
- 参考旧版时，必须先提炼规则、状态、边界与数据结构，再按新架构重写。
- 优先迁移“机制与约束”，而不是机械搬运“实现细节与历史耦合”。
- 如果确实需要借鉴旧代码，默认只抽取最小必要逻辑，并在新文件中重新组织。
- 不要把旧版的大函数、大对象、大段事件逻辑原封不动塞进新项目。
- 任何迁移都要以“更清晰、更可测、更可维护”为判断标准，而不是“复制得更快”。

## 模块完成后的回查要求
- 每完成一个模块，必须回到旧代码再次核对：
  - 设计是否一致
  - 逻辑是否一致
  - 数值是否一致
  - 是否遗漏关键边界条件
  - 是否把旧版冗余一并错误带入新框架
- 若发现偏差，应优先修正设计偏差，而不是先继续堆下一个模块。

## 低耦合重构目标
- 新框架必须优先保证低耦合、易修改、易删除。
- 存档、成就、角色选择、数据库、事件、关系、商店等系统，不得互相硬耦合成不可拆结构。
- 设计时必须考虑：未来用户可能大幅修改、替换、甚至删除某一整块系统。
- 默认做法是：
  - 状态层与规则层解耦
  - 规则层与 UI 表现解耦
  - 持久化与核心流程解耦
  - 可选系统通过清晰接口接入，而不是散落地嵌进核心流程

## 默认执行方式
- 在任务方向已明确时，默认连续自主执行，不要频繁停下来等待用户说“继续”。
- 不要因为普通的小决策、小改动、小范围实现而反复征求确认。
- 能通过现有上下文做出低风险合理假设时，直接继续推进，并在阶段性汇报时简要说明假设。
- 默认按“审计 → 计划 → 实现 → 验证 → 回查 → 下一轮”的循环持续工作，直到完成当前阶段目标，或出现明确阻塞。
- 只在以下情况才主动停下并向用户汇报或提问：
  - 遇到真实阻塞，无法靠本地信息继续推进。
  - 发现需求冲突、范围失控或高风险方向错误。
  - 即将执行明显破坏性、不可逆或超出当前主线的大操作。
  - 当前阶段目标已经完成，且下一阶段方向不明确。
- 普通进度不需要高频汇报；仅在关键里程碑、重大风险或阶段完成时再简洁同步。

## 长时间运行偏好
- 默认允许进行长时间、连续、多轮的自主迭代，不要人为把工作切成大量需要用户确认的小段。
- 若任务仍在主线范围内且没有新阻塞，应继续向前推进，而不是提前收尾。
- 不要因为“已经做了一小时左右”就结束；默认应继续多轮推进，直到当前阶段真正完成。
- 一次工作中可以连续完成：旧版审计、方案细化、脚手架搭建、模块实现、局部验证、回查旧版、修正回归、进入下一轮。
- 如果验证失败，优先自行定位和修复；不要一遇到报错就停止。
- 在不偏离主线的前提下，允许连续进行数小时级别的重构迭代。

## 自动停止条件
- 如果当前主线仍然明确、仍可继续实现、且没有出现新阻塞，则不得因为“已经做了一些内容”就提前停止。
- 如果测试或构建失败，应先自行定位、修复并再次验证，除非失败原因超出当前上下文可解范围。
- 如果还有同一阶段内明显相邻的下一步，应直接进入下一步，而不是把流程控制权交还给用户。
- 只有在“当前阶段目标完成”“出现真实阻塞”“出现高风险方向不确定性”三类情况之一发生时，才允许结束本轮连续工作。

## agent team 使用规则
- 应积极、合理地使用 agent team，而不是所有事情都由主代理串行完成。
- 旧版审计阶段尤其应优先使用 explorer 并行拆分：
  - 角色 / 导师 / 学位 / 毕业线
  - 论文 / 投稿 / 审稿 / 科研分
  - 事件系统 / 分支 / 触发条件
  - 存档 / 成就 / 数据库 / 统计 / 其他耦合系统
- 主代理负责：确定主线、拆分任务、控制边界、整合结果、做最终决策。
- 子 agent 任务必须边界清晰、职责单一、写入范围尽量不重叠。
- 若多个子 agent 并行工作，主代理应避免重复劳动，并在等待期间继续做不冲突的本地工作。
- 对阻塞主线的关键任务，优先由主代理直接处理；对可并行的侧任务，优先交给子 agent。
- 不要为了“用了 agent team”而生造拆分；只有在能明显提速、降风险或并行推进时才拆分。

## 工作边界与顺序
当前阶段默认按以下顺序推进：
1. 审计旧版真实设计与逻辑。
2. 甄别冗余、弃用、重复与耦合点。
3. 形成详细重构计划与模块边界。
4. 搭建或调整新架构以承载已确认规则。
5. 逐模块重构，每完成一块就回查旧版。
6. 再逐步扩展事件、关系、商店、成就、结局、存档等外围系统。

## 重构原则
- 先做骨架，再补内容；先做核心，再补外围。
- 小步推进，持续验证，避免一次性铺太多系统。
- 优先确保架构清晰，而不是短期功能数量最大化。
- 默认不为了兼容历史临时结构而牺牲新架构质量。
- 若新版本与旧版玩法不一致，必须区分“有意重设计”与“尚未迁移完成”。
- 不默认机械搬运旧版全部代码；优先迁移规则，再迁移内容和表现层。

## 沟通偏好
- 回答保持直接、简洁、可执行。
- 先给结论，再给必要理由。
- 当“继续实现”和“先定方案”都可行时，优先选择更有利于持续推进主线的一边。
- 不要频繁把流程控制权交还给用户；默认由代理持续推动当前主线。

## 记忆摘要
- 这是一个从零开始的新重构项目。
- 现在最重要的不是恢复旧目录，而是建立正确的新架构。
- 现在最重要的不是同时做很多东西，而是持续推进唯一主线。
- 默认主线就是：新的游戏本体重构。
- 默认工作方式就是：连续自主迭代，遇到真正问题再停下。
- 默认重构方式就是：先完整审计旧版，再抽象，再重写，而不是直接复制旧版实现。
